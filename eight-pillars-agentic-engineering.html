Eight Pillars of Agentic Software Engineering

A practical framework for reliable, observable and governed coding-agent systems — with an implementation assessment of Claude Code.

Claude Code Harness engineering Verification Governance Multi-agent

Central thesis

Reliable agentic software engineering comes from placing a capable model inside a stateful, legible and bounded delivery system with short executable feedback loops, enforceable validation and one accountable coordinating authority.

01 — Problem statement

The model is not the system

Frontier models write strong code. Model capability alone does not reliably produce correct, maintainable software over long-running tasks. The agent must interpret ambiguous intent, inspect unfamiliar repositories, manage finite context, recover from failures and prove that behavior meets acceptance criteria.

Coding assistance

  • Produces or explains code in small increments
  • Developer holds all state
  • Every output is read immediately
  • Failure is visible and cheap

Agentic coding

  • Inspects files, runs commands, edits the repository, launches tests
  • Continues working with limited intervention
  • Preserves state across sessions and context boundaries
  • Can assert success without sufficient evidence

A powerful model without supporting structure may generate plausible code while losing the original requirement, relying on stale assumptions, stopping at partial completion, or declaring success it cannot demonstrate. The surrounding harness — how the model receives instructions, selects context, invokes tools, maintains state, requests approval, delegates, validates and reports — is therefore part of the engineering system.

Which model writes the best code?

Which model, harness, repository and execution environment most reliably turns intent into verified software changes within our risk, cost and review constraints?

02 — Design principles

Six commitments behind the eight pillars

2.1

The model is one component

Performance depends on tool schemas, repository legibility, context selection, isolation, feedback and stopping conditions. Two products on similar models behave differently because their harnesses expose different information and enforce different boundaries.

2.2

Ground truth comes from the environment

The agent should not trust its own narrative of what it changed. Inspect files, run commands, read exit codes, review diffs, compare behavior with acceptance criteria. Claims of completion stay grounded in observable state.

2.3

Artifacts beat conversational memory

Plans, task ledgers, commits, test results and handoff notes provide durable continuity. They reduce dependence on the model reconstructing decisions from a long or compressed transcript.

2.4

Autonomy scales with verification capacity

More autonomy helps only when the system can detect mistakes and a responsible human can assess the evidence. Throughput beyond review capacity increases risk, not productivity.

2.5

Isolation and concurrency are separate decisions

Isolation is broadly useful for safety, clean state and conflict reduction. Concurrency is conditional — it pays off only when tasks are genuinely independent and integration can be reviewed coherently.

2.6

One authority stays accountable

Agents can plan, implement, test and review. Consequential changes still require an identified authority responsible for synthesis, acceptance and release. Today that authority should normally be human.

04 — How the pillars work together

Eight pillars, one cycle

The pillars are a connected delivery system, not eight independent practices. Expressed as a loop, the framework runs in a single repeating order.

  • P2Specification defines the desired state and acceptance criteria.
  • P2Unit-of-work design converts the objective into tractable increments.
  • P3Context engineering supplies the information the current increment needs.
  • P4Tool-enabled execution lets the agent inspect and change the real environment.
  • P1The stateful loop uses observed results to decide what to do next.
  • P5Verification determines whether the increment is complete.
  • P6The coordinating authority resolves ambiguity and accepts the outcome.
  • P7Isolation and conditional concurrency control how work is separated and scaled.
  • P8Observability and boundaries keep the process reviewable, affordable and safe.

The system is only as strong as its weakest control. Excellent tools cannot compensate for an unclear requirement. Strong tests cannot guarantee maintainability. Parallel agents cannot compensate for poor decomposition. Detailed instructions cannot enforce security boundaries. Human review cannot scale if the harness produces unstructured output faster than people can understand it.

08 — Conclusion

A systems discipline, not faster typing

Agentic software engineering is not conventional development performed faster by a language model. It is a distinct systems-engineering discipline centered on the harness, execution environment, context strategy, verification architecture and human operating model surrounding the model.

  • 01The stateful reconciliation loop keeps reasoning grounded in desired and observed state.
  • 02Specification and unit-of-work design make substantial work tractable.
  • 03Context and knowledge engineering preserve the right information without flooding the context.
  • 04The tool-enabled execution environment connects reasoning to real software systems.
  • 05Harness-enforced verification makes completion evidence-based.
  • 06A single accountable authority preserves coherence and ownership.
  • 07Isolation with conditional concurrency scales safely without assuming parallelism.
  • 08Observable, economical and bounded operation makes autonomy governable.

The practical objective is not maximum autonomy. It is the highest level of autonomy that remains verifiable, reviewable, economical and accountable.
