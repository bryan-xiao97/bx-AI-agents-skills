Add a TechCrunch RSS source to news-digest

Project
news-digest
Date
June 29, 2026
Re
Adding TechCrunch as a third feed
Source
Direct request — no PRD on file; priorities confirmed interactively

Adding TechCrunch is a small change: it is plain RSS, and the existing Ars Technica module already handles its exact shape. The one decision worth making — given a "maintainability first" priority — is to extract a shared RSS core rather than clone a third near-identical module. Content depth is settled as excerpt-only, which the live feed makes the only realistic option anyway.

Priorities driving trade-offs

Maintainability first (this codebase will keep accruing sources — keep source #4/#5 cheap), then correctness, then minimal change. Content depth is fixed: excerpt-only, matching Ars Technica, with no Jina enrichment for TechCrunch.

Verified fact — 2026-06-29

https://techcrunch.com/feed/ is a WordPress RSS 2.0 feed. Each <item> carries title, link, a stable <guid isPermaLink="false">…?p=NNN</guid>, and a single-sentence <description> (CDATA). There is no <content:encoded>. The existing arstechnica._extract_content already handles this: it prefers content, falls back to summary (the description), strips HTML, and caps at 2000 characters.

Design problem: clone the Ars module, or generalize the RSS source

The mechanical work has only one sensible shape and is spec'd below without ceremony: a TechCrunch source that normalizes feed entries to Article (id="techcrunch:<guid>", skip entries missing title/link, stable_hash fallback), a techcrunch_top_n setting, a guarded fetch in main._gather_articles, and a mirrored test module. The one consequential fork is how to structure the now-duplicated RSS code.

At stake: whether the codebase carries two (soon N) near-identical RSS modules each duplicating _TextExtractor, _strip_html, and _extract_content, or a single shared RSS core. This is the only decision here that is expensive to reverse, and it is exactly the axis the "maintainability first" priority targets.

Candidate approaches

  • A — Clone: copy arstechnica.py to techcrunch.py verbatim, swap the feed URL and the id/source strings. Pros: smallest, lowest-risk diff; proven code; Ars untouched. Cons: helpers now live in two files; a fix must be made in N places; source #4 repeats the duplication. Directly contradicts the priority.
  • B — Shared core + wrappers: move the helpers plus a generic fetch_rss_feed(client, feed_url, *, source, limit) into sources/_rss.py; arstechnica.py and techcrunch.py become ~3-line wrappers. Pros: one home for RSS parsing; source #4 is a wrapper; per-source seam survives for quirks; the Ars refactor is covered by existing tests. Cons: touches the working Ars module (small, test-guarded); slightly more upfront work.
  • C — Data-driven: one fetch_rss_feed plus a config list of (source, feed_url, top_n) iterated in main. Pros: maximum DRY; a new source is one config line. Cons: collapses the per-source seam; the per-source try/except isolation in _gather_articles must also be generalized — more churn for marginal gain. Over-engineered for two RSS sources.
Trade-offs across the dimensions that discriminate
Dimension A — Clone B — Shared core C — Data-driven
Duplication / cost of source #4 High — copy whole module Low — 3-line wrapper Lowest — one config line
Upfront effort & risk Lowest — Ars untouched Low — Ars refactor, test-guarded Medium — also generalize error isolation
Per-source flexibility High (separate files) High (wrapper seam) Low (uniform loop)
Regression risk to Ars path None Low (covered by tests) Medium
Fit to "maintainability first" Poor Strong Strong but overshoots

Recommendation — Approach B

Extract a shared sources/_rss.py core and make Ars and TechCrunch thin wrappers over it. It is the direct expression of the priority: it removes the duplication cloning would entrench while keeping a per-source seam for future quirks. The one real cost — refactoring the working Ars module — is bounded by the existing test_arstechnica.py suite, which exercises every helper being moved; run it before and after and any regression surfaces immediately. C is rejected as premature: with two sources, the per-source try/except isolation is a feature, not boilerplate. This rests on one assumption — that the refactor preserves the public fetch_feed signature so main.py is unaffected; the wrapper makes that cheap to honor.

System architecture

No architectural change — TechCrunch slots into the existing fetch → dedup → summarize → render → send → archive pipeline as a third source. The only structural change is internal to the sources/ package: RSS parsing is consolidated behind _rss.py.

sources/ ┌───────────────────────────────────┐ [GitHub Actions cron] → │ hackernews.py ─┐ │ main.run() │ arstechnica.py ┤→ _rss.py (shared) │ │ │ techcrunch.py ─┘ (RSS core) │ ▼ └───────────────────────────────────┘ _gather_articles() ── fetches all 3 sources, each guarded by its own try/except │ ▼ filter_new() ──→ summarize_all() ──→ render_digest() ──→ send_email() ──→ Archive (dedup) (Gemini themes) (html / text) (Gmail SMTP) (Postgres)

Data flow

  1. main.run opens the repo, loads the dedup ledger, opens one shared httpx.AsyncClient.
  2. _gather_articles fetches HackerNews, then Ars, then TechCrunch — each in its own try/except so a TechCrunch outage logs and continues.
  3. techcrunch.fetch_feed → _rss.fetch_rss_feed GETs and parses the feed, and for each of the top techcrunch_top_n entries: skips items missing title/link, derives id="techcrunch:<guid>" (stable-hash fallback), extracts the description excerpt (capped at 2000 chars), returns Article(source="techcrunch", score=None).
  4. filter_new drops any TechCrunch id already in the ledger window.
  5. summarize_all (one Gemini call) assigns each surviving article a summary, theme, and relevance.
  6. render_digest groups by theme (TechCrunch intermixes with the rest); send_email delivers; save_sent archives the techcrunch:<guid> ids.

Integration points & dependencies

System / API Direction Description
TechCrunch RSS (techcrunch.com/feed/) In Public WordPress RSS 2.0; no auth, no secret, no rate-limit concern at one fetch/day.
HackerNews / Ars Technica In Unchanged existing sources.
Google Gemini, Gmail SMTP, Postgres Both / Out Unchanged; TechCrunch articles flow through them like any other.

Upstream (this relies on): feedparser, httpx — already present, no new dependency; plus TechCrunch feed availability and guid stability. Downstream (relies on this): the source-agnostic _gather_articles → summarize_all → render_digest → archive chain, which needs no change beyond the one fetch call.

Non-functional requirements

Requirement Target
Failure isolation A TechCrunch fetch error logs via logger.exception and the digest still sends from the other sources.
Fetch latency Completes within the existing 60s AsyncClient timeout; a single GET + parse, no per-item enrichment.
Prompt-size safety Each excerpt capped at MAX_CONTENT_CHARS (2000) by the shared _extract_content.
Dedup correctness Stable techcrunch:<?p=NNN> ids prevent re-sends across the dedup window.
Test coverage Project gate of ≥80% maintained; new modules covered; Ars suite stays green through the refactor.

Risks and mitigations

Risk Mitigation
Refactoring the working Ars module regresses RSS parsing Move helpers behind the existing public fetch_feed signature; run test_arstechnica.py before and after. Land the refactor and the TechCrunch-add as two separate commits to isolate blame.
TechCrunch gives Gemini a thinner signal Accepted per the excerpt-only decision: only a one-sentence description, no content:encoded. Summaries fall back to title+excerpt. Revisit with Jina enrichment only if summaries prove weak.
TechCrunch has no HN-style score, so "Also Notable" can't gate it on unthemed_min_hn_score Unthemed TechCrunch items are kept on relevance alone — same path as Ars, which also has no score. Existing build_groups behavior; noted so the mix shift is expected, not a bug.
Feed volume floods the digest (TechCrunch publishes hourly) Cap with techcrunch_top_n (default 5, matching ars_top_n); the dedup window prevents re-sends.
guid format changes upstream, breaking dedup stable_hash(title, link) fallback covers a missing/blank guid; a format change is low-probability and self-heals next run.
