Scope assumption this design rests on

This surface is demo-only — it is not intended to receive real client data. That is what makes the free tier acceptable: on the free tier Google may use prompts and responses to improve its products, which is moot when no MNPI is present. Guardrail: if real deal data is ever introduced, revisit the provider tier first — moving to the paid Gemini API or Vertex (where that stops) is a key/endpoint swap, not a code rewrite.

Context

How “Generate email” works today

  • The button (BuyerRequests.tsx:100) is gated by genEnabled — enabled only when every question in a conversation is resolved and at least one is approved (mockStore.ts:79).
  • Click → DealWorkspace.handleGenerate → api.generateEmail(dealId, convId) → returns an EmailDraft → opens EmailModal for review/edit → “Send to my inbox.” The platform never emails the buyer; the banker sends from Outlook.
  • api.generateEmail (api.ts:96) already has two backends: the in-memory mock and a real branch that POSTs to /api/deals/{id}/conversations/{id}/email. That Azure Function does not exist yet.
  • There is no LLM today. composeEmail (mockStore.ts:100) is deterministic string templating: filter to approved, prefer the banker’s edit else stripCitations(answer), concatenate under a fixed greeting/sign-off.

Binding constraints — from CLAUDE.md & the codebase

These bind every choice below

  • No business logic in components; api.ts is the only data path. The LLM call therefore lives server-side, mirrored by the mock — never in the SPA.
  • “Approved-only and citations stripped — enforced at composition, not by the prompt.” Asserted in mockStore.ts and the EmailModal banner. The model is never the thing trusted to filter or strip.
  • Content is confidential buyer diligence (revenue, customer concentration, litigation, HIPAA status). Data governance is first-class.
  • A human is always in the loop; nothing is auto-sent — this caps the blast radius of a bad generation.

Provider tier & data boundary

At stake: whether confidential deal data ever transits an endpoint that may use it for training — a hard-to-reverse, reputational mistake. Forced by the “free tier” choice colliding with the sensitivity of the answers.

One free key, no billing, dev + prod.

Zero setup, no card, no cost.

BFree (dev) → Paid Gemini API (prod)

Same SDK, same model id; flip billing on and swap the key. The escalation path if real data is ever introduced.

Paid tier does not train on your data; cost stays sub-cent; higher limits.

A billing + key-rotation step; unneeded while the surface stays demo-only.

CVertex AI (prod)

Same model via Google Cloud with region pinning, VPC-SC, IAM federation.

Strongest governance — residency, network isolation, enterprise IAM.

Heaviest setup; most at odds with the cost/ops-simplicity priority.

Recommendation

AFree AI Studio tier, in dev and prod

The surface is scoped to synthetic/demo content and is not intended to receive real client data, so the free tier’s data-for-training term doesn’t bind — the MNPI-governance dimension is moot. With that off the table, A wins outright on the stated priority: zero setup, no billing account, no spend; the tighter rate limits are a non-issue for human-driven clicks. Recorded assumption: no real deal data (MNPI) ever flows through Generate-email — if that changes, escalate to B (paid) or C (Vertex), a key/endpoint swap, not a code rewrite.

Where the Gemini call executes

At stake: API-key secrecy and where the approved-only + citation-strip guardrails physically run. Forced by “no business logic in components / api.ts is the only data path” and the “not by the prompt” rule.

AServer-side Azure FunctionRecommended

The existing POST /api/…/email route holds the key, filters/strips, calls Gemini, returns an EmailDraft.

Key never reaches the browser; guardrails in a trusted boundary; zero SPA change.

Requires standing up the Function (doesn’t exist yet).

BBrowser-directDisqualified

SPA calls Gemini with a key in a VITE_* env var.

No backend — superficially the most “minimal.”

VITE_* is baked into the bundle: key visible in devtools, no spend ceiling. Guardrails would run in untrusted client JS — violates both binding rules.

CLocal proxy (dev) + Function

A Vite dev-middleware or local Functions host serves the same route for local iteration.

Exercise the real LLM path without cloud infra; identical contract to prod.

A little dev plumbing.

B is off the table on key exposure alone, regardless of the simplicity pull. The server endpoint already exists as a contract in api.ts; we implement the Function behind it. The mock path stays LLM-free so unit tests remain deterministic and offline — and it doubles as a documented “basic draft” fallback.

03

Guardrail integrity under synthesis

At stake: with the model now writing prose, a buyer-facing email could re-introduce [cN] markers, include a rejected answer, or invent or distort a figure. Forced by the synthesize scope against the “enforced at composition” rule. All approaches keep the same mandatory input layer: code filters to approved + strips citations before the prompt — the model never sees rejected items or markers.

APrompt-only + review

Grounding instructions plus the banker’s review.

Cheapest, one call.

No deterministic backstop if the model emits a stray [c1].

BPrompt + deterministic post-processRecommended

Re-run stripCitations() on the output; validate shape; error on failure.

Citation-stripping stays a code guarantee on input and output. Cheap.

Faithfulness is mitigated, not guaranteed — see note.

C+ LLM-judge verification

A second call compares draft vs. source for unsupported claims.

Strongest automated faithfulness check.

Doubles cost/latency + orchestration; marginal given the human gate.

Trade-offs
DimensionA — Prompt onlyB — Prompt + post-process✓ recommendedC — + LLM judge
Citation-leak guaranteePrompt-dependentDeterministicDeterministic
Faithfulness assuranceReview onlyPrompt + reviewPrompt + review + judge
Cost / latency1 call1 call2 calls
Fit to cost-simplicityGoodBest balancePoor
Recommendation

BDeterministic filter/strip on both sides; review for faithfulness

Keep the input filter (mandatory), re-apply stripCitations() to the output as belt-and-suspenders, validate shape, and use a grounding prompt at low temperature (~0.2). Honest limitation: faithfulness of synthesized prose is the one property code can’t enforce deterministically — the controls are the grounding prompt, low temperature, and the mandatory banker review before any send. Defer C until real-data experience shows review is insufficient.

04

Call shape & failure UX

At stake: complexity vs. perceived latency, and how the error reaches the banker. Forced by the failure decision (“show the error in the UI”) against the existing button → modal flow.

ASynchronous request/responseRecommended

Button shows a loading state, awaits the full draft, opens the modal.

Zero transport change; matches today’s await api.generateEmail; Flash returns in seconds.

Button spins a few seconds with no partial output.

BStreaming (SSE)

Modal opens immediately; tokens stream in.

Better perceived latency.

New transport + incremental render + harder abort/error — for a draft acted on only when complete.

Trade-offs
DimensionA — Synchronous✓ recommendedB — Streaming
Transport / contract changeNoneNew (SSE)
Modal rendering changeAdd loading + error stateIncremental render
Perceived latencyA few secondsBetter
Fit to existing flow + costBestPoor
Recommendation

ASynchronous, with a loading and error state

Keep the synchronous contract. Add a loading state on the button and a client timeout (~30 s). On failure the Function returns a structured { status, message }; api.ts already throws ApiError (api.ts:46) — extend the DealWorkspace handler to render an error state with the message + Retry. EmailModal renders only when a draft exists, so the error UI belongs in the generate handler, not inside the modal.

System architecture

The path of a generated draft

The SPA contract is unchanged — api.generateEmail still returns Promise<EmailDraft|null>. A new server-side Function does the real work; the two teal steps are the deterministic guardrails that keep the “enforced at composition, not by the prompt” guarantee intact.

1
SPA → api.ts → POST /api/…/email

Button (loading state) → handleGenerate → api.generateEmail; Entra bearer token attached. Mock branch stays deterministic for tests/offline.

2
Function re-checks the approval gate

resolved == total && approved > 0. Gate closed → returns 200 null (mirrors composeEmail).

3
Deterministic filter — approved only, citations stripped

Prefer the banker’s edit, else stripCitations(answer). This is the only content the model is ever allowed to see.

4
Build the grounded prompt

Question + clean-answer pairs, the contact, the house tone. Answer text is treated as data, not instructions.

5
Call Gemini 2.5 Flash

Key from Function config / Key Vault. Temperature ~0.2 to curb drift.

6
Deterministic post-process — strip + validate

stripCitations() on the output (belt-and-suspenders) + non-empty / length / sign-off checks.

7
Assemble & return the EmailDraft

{ to, subject, body, approvedCount } → 200 → EmailModal opens for review → banker sends from Outlook.

!
Any failure → structured error

{ status, message } → ApiError in the SPA → error state with message + Retry. Client timeout ~30 s.

gate / transport deterministic guardrail failure path

Data flow

  1. Banker resolves all questions; the gate opens (mockStore.ts:79) and the button enables.
  2. Click → handleGenerate sets loading → api.generateEmail(dealId, convId).
  3. api.ts (real branch) POSTs to the Function with the Entra bearer token.
  4. Function re-checks the gate; if closed, returns 200 null.
  5. Function filters to approved, prefers the banker’s edit, else strips citations — the model’s allowed input.
  6. Function builds the grounded prompt and calls Gemini 2.5 Flash (temp ~0.2).
  7. Function strips citations on the output, validates, assembles the EmailDraft.
  8. Success → modal opens. Failure → ApiError → error state + Retry.
  9. Banker edits → “Send to my inbox” → reviews and sends from Outlook (platform never sends).
